splonks-cpp literal port caveats
================================

Purpose:
- record every non-literal compromise made during the port
- keep these explicit so they can be revisited later
- avoid "temporary" substitutions disappearing into the codebase

Current caveats
---------------

1. SID spatial index backend
   - Rust source: `splonks-rs/src/sid.rs`
   - Current C++ port: `splonks-cpp/src/sid.cpp`
   - Caveat:
     Rust uses `rstar::RTree`.
     The current C++ port uses a linear scan over stored `SIDNode`s.
   - Why:
     Preserve the visible query behavior first without blocking the port on
     choosing and integrating a spatial index library.
   - Future plan:
     Swap the internal backend to a real spatial index later without changing
     the outward `SID` API.

2. `glam` vector types
   - Rust source usage: broad, currently first used in `direction/utils/sid`
   - Current C++ port: `splonks-cpp/src/math_types.hpp`
   - Caveat:
     Rust uses `glam::Vec2` and `glam::IVec2`.
     The current C++ port uses a tiny local shim with `Vec2` and `IVec2`.
   - Why:
     Preserve the shape of the port and keep type usage explicit before pulling
     in a full math library.
   - Future plan:
     Replace or adapt this shim to `glm` later once the port has enough surface
     area to justify the dependency.

3. `SpecialEffect` trait macro pattern
   - Rust source: `splonks-rs/src/special_effects/special_effect.rs`
   - Current C++ port: `splonks-cpp/src/special_effects/special_effect.hpp`
   - Caveat:
     Rust uses a trait plus the exported `special_effect_getters!` macro to stamp
     out shared getter implementations on concrete effect types.
     The current C++ port uses a plain abstract interface plus free helper
     functions for sample-region lookup.
   - Why:
     There is no direct C++ equivalent to that Rust trait + exported macro shape
     that is worth preserving literally.
   - Future plan:
     Keep the shared behavior explicit and boring in C++; if repeated getter code
     becomes noisy, introduce a tiny helper base struct later instead of a macro.

4. `Origin` peeled out of `entity.rs`
   - Rust source: `splonks-rs/src/entity.rs`
   - Current C++ port: `splonks-cpp/src/entity_origin.hpp`
   - Caveat:
     `sprite.rs` needed `Origin`, but porting all of `entity.rs` this early would
     drag in a huge non-leaf module.
   - Why:
     Keep the near-leaf pass moving without fabricating the rest of `entity.rs`.
   - Future plan:
     Fold this back into the real `entity` port once `entity.rs` is translated.

5. Sprite JSON loading backend
   - Rust source: `splonks-rs/src/sprite.rs`
   - Current C++ port: `splonks-cpp/src/sprite.cpp`
   - Caveat:
     Rust uses `serde_json`.
     The current C++ port uses local `nlohmann::json`.
   - Why:
     The workspace already vendors `nlohmann/json.hpp`, and this keeps the port
     behaviorally close without adding a new dependency.
   - Future plan:
     Keep `nlohmann::json` unless a better reason appears to swap it.

6. Audio playback backend
   - Rust source: `splonks-rs/src/audio.rs`
   - Current C++ port: `splonks-cpp/src/audio.cpp`
   - Caveat:
     Rust uses raylib music/sound objects that can load and play `.ogg` files
     directly.
     The current C++ port preserves the `Audio` surface API, validates that the
     asset files exist, and stores playback state, but does not yet decode or
     play audio.
   - Why:
     SDL3 alone does not give us literal `.ogg` playback parity for this module
     without choosing and integrating an additional decoder/playback layer.
   - Future plan:
     Swap the internal backend to SDL3 + a real decoder/playback path later
     without changing callers.

7. Entity display mapping support slice
   - Rust source: `splonks-rs/src/entity_display_states.rs`
   - Current C++ port: `splonks-cpp/src/entity_display_states.cpp`
   - Caveat:
     Rust takes a full `Entity`.
     The current C++ port takes a temporary `EntityDisplayInput` carrying only
     `type_` and `display_state`.
   - Why:
     This keeps the display-state mapping literal without dragging the entire
     `entity.rs` monster into the port yet.
   - Future plan:
     Fold the support slice back into the real `Entity` port once `entity.rs`
     lands.

8. Entity core support header
   - Rust source: `splonks-rs/src/entity.rs`
   - Current C++ port: `splonks-cpp/src/entity_core_types.hpp`
   - Caveat:
     The full `Entity` struct is still not ported, but many modules need its
     enums, constants, and identity types.
   - Why:
     Centralizing the peeled shared pieces avoids a pile of ad hoc one-off
     support headers and unlocks the rest of the literal port incrementally.
   - Future plan:
     Keep migrating shared `entity.rs` definitions into the real `entity` port
     and retire the temporary split once `entity.hpp/.cpp` exist.

9. `JUMP_DELAY_FRAMES` peeled from `player.rs`
   - Rust source: `splonks-rs/src/entities/player.rs`
   - Current C++ port: `splonks-cpp/src/entity.hpp`
   - Caveat:
     `Entity::new()` depends on `JUMP_DELAY_FRAMES`, but porting all of
     `player.rs` just for that constant would be stupid at this stage.
   - Why:
     Keep the real `entity` port moving.
   - Future plan:
     Replace the temporary `kJumpDelayFrames` with the actual shared player
     constant once `player.rs` is ported.

10. Partial `Stage` port before room generation
   - Rust source: `splonks-rs/src/stage.rs`
   - Current C++ port: `splonks-cpp/src/stage.cpp`
   - Caveat:
     `State` and collision helpers needed the `Stage` type and several tile query
     methods, but the real `Stage::new()` path still depends on unported
     `room.rs` and stage generation code.
   - Why:
     This unlocks `state.rs` and the growing world/state layer without inventing
     fake shapes for those callers.
   - Future plan:
     Finish the full `stage.rs` translation once `room.rs` and `stage_gen/*`
     exist, then replace the temporary `rooms` placeholder and missing generator
     path with the real layout/build logic.

11. Partial `player.rs` port
   - Rust source: `splonks-rs/src/entities/player.rs`
   - Current C++ port: `splonks-cpp/src/entities/player.cpp`
   - Caveat:
     The C++ file currently ports the player constants and `set_entity_player`,
     but not yet the large logic/physics stepping functions.
   - Why:
     `stage_init.rs` needed the real player constructor path first, and that is
     a clean incremental landing without fabricating the rest of player control
     and combat behavior.
   - Future plan:
     Port the remaining player logic and physics functions into the same file
     and then retire this caveat.

12. `step.rs` / `step_entities.rs` signature and dispatch are partial
   - Rust source: `splonks-rs/src/step.rs`, `splonks-rs/src/step_entities.rs`
   - Current C++ port: `splonks-cpp/src/step.cpp`, `splonks-cpp/src/step_entities.cpp`
   - Caveat:
     The Rust files take raylib handle/thread parameters and dispatch into the
     full entity module set.
     The current C++ port collapses the step signatures down to `State`,
     `Audio`, `Graphics`, and `dt`, and only the player + common call paths are
     currently wired in `step_entities.cpp`.
   - Why:
     This preserves the real step loop structure and comments now, without
     pretending the missing entity modules already exist.
   - Future plan:
     Rewire these files onto the eventual SDL-side runtime/context and fill in
     the remaining entity dispatch branches as each entity module lands.

13. `entities/common.rs` is only partially live
   - Rust source: `splonks-rs/src/entities/common.rs`
   - Current C++ port: `splonks-cpp/src/entities/common.cpp`
   - Caveat:
     The common file now carries real stun, animation, euler, gravity, grounded,
     damage, jump/climb, tile/entity collision, hurt-on-contact, projectile
     hurt, spikes, death/deactivate, and grounded-on-entities helpers. Some of
     the later Rust helper inventory such as explosion/fall helpers is still not
     surfaced in C++.
   - Why:
     This unlocks real gameplay support under `player.rs` without pretending the
     rest of the entity/collision stack already exists.
   - Future plan:
     Keep pulling the remaining literal Rust helper inventory across as the
     dependent entity files land.

14. SDL runtime/render bridge is still a temporary parity layer
   - Rust source: `splonks-rs/src/render.rs`, `splonks-rs/src/render_tiles_and_entities.rs`,
     `splonks-rs/src/render_ui.rs`, `splonks-rs/src/render_debug.rs`,
     `splonks-rs/src/menu_title.rs`, `splonks-rs/src/menu_settings.rs`,
     `splonks-rs/src/menu_video.rs`, `splonks-rs/src/inputs.rs`
   - Current C++ port: `splonks-cpp/src/render*.cpp`, `splonks-cpp/src/menu_*.cpp`,
     `splonks-cpp/src/inputs.cpp`, `splonks-cpp/src/app.cpp`
   - Caveat:
     The current C++ port now has real SDL-side input dispatch, menu state
     machines, runtime stepping, and the render stack split back into the same
     file boundaries as Rust. The rendering path is still using SDL debug text
     and primitive rectangles instead of the raylib texture/shader pipeline, and
     `App` falls back to empty `Graphics` / `Audio` instances when the current
     asset set is incomplete.
   - Why:
     This gets menus, stage transition, HUD values, and basic playing visuals on
     screen now without blocking the port on full texture/audio backend parity.
   - Future plan:
     Replace the primitive SDL render bridge with a literal-enough texture/shader
     path and remove the `Graphics` / `Audio` fallback bootstrapping once the
     asset/backend layer is complete.
